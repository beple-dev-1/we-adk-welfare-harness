# 복지AX 운영 장애 방지 안티패턴

실제 서비스 운영 중 발생 가능한 장애 패턴을 정리한다.
코드 리뷰 시 IC(차단) 항목은 CRITICAL, IW(권고) 항목은 WARNING으로 처리한다.

---

## IC — 차단 (CRITICAL)

### IC01: 복지혜택 지급·취소 API 중복 처리 방지 미적용

**패턴**: 동일한 지급 요청이 네트워크 재시도나 UI 더블클릭으로 중복 실행될 때 처리하지 않음  
**증상**: 동일 회원에게 혜택이 2회 이상 지급되거나 중복 취소 발생  
**탐지**: 지급/취소 API에 멱등성 키 파라미터 또는 unique 제약이 없는 경우  
**조치**: `idempotency_key` 컬럼 + unique 제약, 또는 DB 락 기반 중복 방지 로직 적용

### IC02: 배치/최종 단계 재검증 누락

**패턴**: 정산 배치에서 지급 여부를 한 번만 확인하고 최종 단계에서 재검증하지 않음  
**증상**: 배치 도중 장애 복구 후 재실행 시 중복 정산  
**탐지**: 배치 Job에 멱등성 체크 및 처리 상태 검증 없는 경우  
**조치**: 배치 실행 전 처리 상태 확인, 처리 완료 표시, 재실행 안전성 보장

### IC03: UI 비동기 더블클릭 방어 누락

**패턴**: 혜택 사용·지급 버튼을 빠르게 2회 클릭 시 중복 API 호출 발생  
**증상**: 동일 요청 2건 이상 서버 도달 → 중복 처리  
**탐지**: 버튼 클릭 핸들러에 disabled 처리 또는 debounce 없는 경우  
**조치**: 버튼 클릭 즉시 disabled, API 응답 후 활성화 (또는 500ms debounce)

### IC04: 동기 Hot-Path 위 Long SQL → 타임아웃 재시도 폭주

**패턴**: 혜택 조회 API 경로에 대용량 집계 쿼리 삽입  
**증상**: 응답 지연 → 클라이언트 재시도 → DB 부하 폭발  
**탐지**: @Transactional 메서드 내 COUNT/SUM/GROUP BY 집계 쿼리 + 인덱스 미사용  
**조치**: 집계는 별도 배치/비동기로 분리, 캐시 활용

### IC05: 잔액·한도 산정 공식 결함

**패턴**: 복지 포인트 잔액 계산 시 동시성 고려 없이 SELECT → 비교 → UPDATE 순차 실행  
**증상**: 동시 요청 시 잔액 음수 또는 한도 초과 지급  
**탐지**: 잔액 차감 시 SELECT FOR UPDATE 또는 낙관적 락 없이 처리하는 경우  
**조치**: SELECT FOR UPDATE 또는 @Version 낙관적 락 적용

### IC06: 대범위 집계 쿼리 → DB 부하 폭발

**패턴**: 날짜 범위 없이 전체 회원 혜택 집계  
**증상**: DB CPU 100%, 전체 서비스 응답 지연  
**탐지**: WHERE 절 없는 COUNT/SUM 또는 과도하게 넓은 날짜 범위 집계  
**조치**: 날짜 범위 제한, 페이징 적용, 집계는 배치로 분리

### IC07: 트랜잭션 내 외부 REST 호출 → DB 커넥션풀 고갈

**패턴**: `@Transactional` 메서드 내에서 외부 결제 API 또는 알림 API 호출  
**증상**: 외부 API 응답 지연 시 DB 커넥션 점유 → 커넥션풀 고갈 → 서비스 전체 장애  
**탐지**: @Transactional 클래스/메서드 내 RestClient/WebClient/외부 HTTP 호출  
**조치**: 트랜잭션 완료 후 외부 호출, 또는 @TransactionEventListener 활용

## IW — 권고 (WARNING)

### IW01: 배치 실패 누적 정리 부재

**패턴**: 정산·만료 처리 배치 실패 시 실패 레코드가 계속 누적되어 재처리 불가  
**조치**: 배치 실패 테이블 관리, 재처리 주기 설정, 최대 재시도 횟수 제한

### IW02: 실패 알림 미정의

**패턴**: 혜택 지급 실패, 정산 오류 발생 시 담당자 알림 경로 없음  
**조치**: 장애 알림 채널 정의 및 Slack/이메일 알림 연동

### IW03: 슬로우쿼리 모니터링 미설정

**패턴**: 운영 환경에서 슬로우쿼리 감지 설정 없음  
**조치**: `spring.jpa.properties.hibernate.format_sql=true`, APM 또는 pg_stat_statements 활성화

### IW04: 가맹점 원거래 확인 미제공

**패턴**: 혜택 취소 시 원거래 가맹점 정보 조회 없이 처리  
**조치**: 취소 API에서 원거래 가맹점 검증 로직 포함

### IW05: JVM 힙 사이즈 근거 부재

**패턴**: 배치 서버 또는 API 서버의 JVM 힙 사이즈가 임의로 설정됨  
**조치**: 실제 부하 테스트 기반 힙 사이즈 산정 및 문서화
